# Phase 6 — Chatbot Knowledge

Source-of-truth for how notes, communications, the timeline, and the AI update rewriter work after Phase 6. Written to stand alone: an LLM with only this file should answer operational questions correctly. Cross-references `02-decisions.md`, `06-business-rules.md`, `08-db-changes.md` but does not depend on them. Builds on Phase 4 (jobs, `job_notes`, `job_events`) and Phase 5 (dispatch, `dispatch_messages`) chatbot knowledge.

## K-6.1 — What Phase 6 adds
Three things. **(1) Communication history:** notes get a visibility classification, and every communication about a job — shared notes, ad-hoc messages, inbound messages, client/vendor updates, dispatch messages — is logged on **one unifying spine** (`communication_logs`), with a delivery state machine. **(2) A rich timeline:** milestones + communications + (curated) notes interleave into one chronological narrative. **(3) The platform's first AI agent:** the **vendor→client update rewriter** turns an internal note into a client-safe draft, which an operator reviews/edits/approves and publishes — running on a reusable **agent substrate** that Phases 7/8/13/16 inherit. Everything the agent does is **draft-then-review-then-publish** (§2.9): the agent never sends to the client, never mutates job/communication state directly.

## K-6.2 — The communication log (the unifying spine)
`communication_logs` is to communications what `job_events` is to milestones: **one row per communication**, denormalized, pointing at its content via polymorphic `source_type` + `source_id` (no FK — spans tables).
- **Columns:** `id`, `tenant_id`, `job_id` (NN), `channel` enum(`internal_note`,`vendor_portal`,`client_portal`,`email`,`sms`,`external_portal`,`phone_call`), `direction` enum(`outbound`,`inbound`,`internal`), `source_type` enum(`dispatch_message`,`outbound_message`,`inbound_message`,`job_note`,`client_update`,`vendor_update`), `source_id`, `visibility` (the shared 5-value enum), `summary` varchar(500) (create-time excerpt for the log/timeline; full body lives in the content row), `sent_by_user_id`, the **recipient** (`recipient_type` enum(`vendor_contact`,`client_contact`,`external`,`internal`,`none`) + `recipient_id` (no FK) + `recipient_email`/`phone`/`cc`/`bcc`), the **delivery layer** (`delivery_status`, `sent_at`/`delivered_at`/`read_at`), `status`, timestamps.
- **The delivery layer lives HERE, not on `dispatch_messages`** — this **supersedes Phase 5 R-5.15**. `dispatch_messages` stays content-only and becomes one `source_type`.
- **Content tables** the spine points at: `outbound_messages` (ad-hoc composed — subject/body/template link), `inbound_messages` (external_sender/raw_body/received_at/parse_status), `client_update_logs` (published rewriter output), plus the existing `job_notes` (share) and `dispatch_messages`. `email_templates` (Mustache `{{...}}`, `applicable_channels` json) exists with no render/send pipeline yet.

## K-6.3 — Note visibility (classification, not sharing)
`job_notes.visibility` is a **5-value** vocabulary, shared app-wide (same enum on `dispatch_messages` and `communication_logs`): `internal_only` (operator workspace, default), `vendor_visible`, `client_visible`, `client_and_vendor_visible`, `requires_review` (flagged for a second look). **Setting a visibility is classification only — it does not share the note** (the explicit-workflow-transitions rule, R-5.8). `NoteVisibilityBadge` renders any value (internal=neutral, vendor=blue, client=green, both=teal, requires_review=amber).

## K-6.4 — Sharing a note → a communication (SHARE-EXISTING)
`shareNote(noteId, audience)` logs a communication **without copying the note** — the spine row has `source_type='job_note'`, `source_id`=the note, and a `summary` excerpt; the full body stays on the note. It's **gated**: a client share needs visibility ∈ {`client_visible`,`client_and_vendor_visible`}; a vendor share needs ∈ {`vendor_visible`,`client_and_vendor_visible`} (else `NOTE_NOT_SHAREABLE`). The comm's **visibility is audience-derived** — a `client_and_vendor_visible` note shared to the vendor yields a **`vendor_visible`** comm (a communication goes to exactly one audience). The recipient is pre-filled (client primary contact; or the single-assignment vendor's primary contact). **Share ≠ Send:** the comm starts at `delivery_status='draft'`. Re-share is allowed.

## K-6.5 — The delivery state machine (Share ≠ Send, Publish ≠ Send)
`delivery_status`: outbound `draft → {sent, queued}`; `queued → sent`; `sent → {delivered, failed}`; `failed → sent`; terminals `delivered`/`bounced`; inbound `received`. Sharing a note and publishing a rewrite draft both create the comm at **`draft`** — neither sends. The operator advances it with delivery buttons that show only legal next steps (the server re-validates every transition). `read_at` is independent of `delivery_status` (a delivered comm can be unread). There is **no external transmission** in Phase 6 — delivery status is operator-tracked; real send is Phase 13.

## K-6.6 — The interleaved timeline + the notes-in-timeline rule
`mergeTimeline(events, communications, notes)` (pure) folds three sources into one chronological list, oldest-first, tie-broken **event < communication < note** (the milestone is the headline of a shared instant). The `<JobTimeline>` UI filters **All / Milestones / Communications / Notes**, groups by day, and uses a **category color axis** (slate=milestone, indigo=communication, rose=note) distinct from the status/visibility/delivery palettes. **A note appears in the timeline iff `(visibility ≠ internal_only)` AND `(it has not been shared as a communication)`** — internal notes stay in the workspace Notes section; a shared note is represented by its communication (showing both would duplicate); the surfaced state is "shareable-but-unshared." A communication appears in **both** the Communications workspace section (with delivery buttons) and the read-only Timeline (the workspace-vs-narrative two-view).

## K-6.7 — The agent substrate (every AI agent writes these — §2.9)
Three generic, **immutable audit** tables (no soft-delete `status`):
- **`agent_runs`** — one row per invocation: `agent_id`, `status`(`running`/`succeeded`/`failed` — the run lifecycle), `trigger_source`, `triggered_by_user_id`, `job_id` (nullable — non-job agents later), `input_summary`/`output_summary` (human-readable excerpts, **not** the source of truth), `model`, `prompt_version`, `input_tokens`/`output_tokens`, `error_message`, `started_at`/`completed_at`.
- **`agent_tool_calls`** — every read/write the agent did, in `sequence`: `tool_name`, `tool_kind`(`read`/`write`), `tool_input`/`tool_output` (json), `status`(`ok`/`error`).
- **`agent_decisions`** — `decision_type`, `proposed_action`, `reasoning`, `confidence`(`high`/`medium`/`low`), `policy_check`, `disposition`(`queued_for_review`/`auto_executed`/`policy_blocked`), `metadata` (json). Phase 6 disposition is always `queued_for_review`.

The shared **runner** (`src/server/agents/runner.ts`): `openRun → registerTool (wraps a tool to auto-log each call) → logDecision → closeRun`. This is the inheritance vehicle for Phases 7/8/13/16. v1 agents use a **fixed pipeline**; the substrate also supports LLM-native tool-use (Phase 8) unchanged. A `registry.ts` (`AGENT_REGISTRY`) enumerates agents with a `testOnly` flag; `listProductionAgents()` hides test fixtures (the committed `test_stub_v1`, an LLM-free substrate-test agent).

## K-6.8 — The update rewriter (`update_rewriter_v1`)
The first production agent. Given an internal **note** + job/client/vendor context, it produces a **client-safe draft** — **stripping** pricing/NTE figures, internal cost/process language, speculation, and vendor names where immaterial; **preserving** what work is happening, timing facts, and next steps; professional tone **without manufacturing certainty** the source doesn't support. It is a **fixed pipeline** (read-broad / write-narrow): read tools `getJobNote` + `getJobDetail` + `listAssignmentsForJob`; one write tool `createRewriteDraft` (a draft at `pending_review`). **Input is polymorphic** (`source_type` on the draft = `job_note` in Phase 6; `vendor_update_log` arrives Phase 10+ with no redesign); **output is always a `client_update_log`** → a `client_portal` communication. The agent is invokable on **any** note regardless of visibility (visibility classifies sharing, not rewriter input). Re-running creates a new draft.

## K-6.9 — The draft lifecycle + the publish path
`update_rewrite_drafts.status`: **`pending_review` → `approved` → `published`** (terminal); `pending_review → rejected` (a formal review with a reason); `pending_review → discarded` (silent dismissal). The operator **reviews** (sees the original note, stripped items, rephrasings, confidence, rationale, and the editable draft), then **approves** (optionally editing — the edit is stored as `update_rewrite_reviews.edited_content`; `draft_content` stays immutable; effective content = `edited_content ?? draft_content`), **rejects**, or **discards**. **`publishRewriteDraft`** is the **only** path from draft to communication: it refuses anything not `approved`, then in one transaction (lock job → lock draft → re-check) writes a `client_update_logs` row + a `communication_logs` spine row (`source_type='client_update'`, `client_portal`, `client_visible`, `delivery_status='draft'`, recipient pre-filled) + advances the draft to `published` + two audit rows. **Publish ≠ Send** — the comm lands at `draft`; the operator Sends it after. A published draft leaves the Update-drafts queue (it *is* its communication now).

## K-6.10 — Policy & the §2.9 invariant (how the agent is bounded)
Phase 6 hardcodes a universal policy: **rewriter drafts always require human review.** This is enforced **structurally**, not by convention: the agent's only write is a `pending_review` draft; it has **no code path** to `communication_logs`, `client_update_logs`, or job status. The `publishRewriteDraft` action (human-initiated) is the sole draft→communication path and gates on `status='approved'`. So a draft is **never auto-published** (acceptance #7 holds because the "auto-publish within policy" branch simply never fires in Phase 6). The `REWRITER_POLICY = { requiresReview: true }` constant is the seam where **Phase 7 per-client `agent_policies`** plug in.

## K-6.11 — LLM integration (routing, model, mock, cost)
Uses the Vercel **AI SDK v6** (`generateObject` with a Zod schema → `{clientFacingText, strippedItems[], confidence, rationale, rephrasings?}`). `resolveRouting()` picks, in precedence: **`REWRITER_MOCK=1`** → deterministic stub (token-free dev/probes, wins over keys); **`AI_GATEWAY_API_KEY`** → gateway, model string `"anthropic/claude-sonnet-4-6"`; **`ANTHROPIC_API_KEY`** → direct `@ai-sdk/anthropic`, model `"claude-sonnet-4-6"`; none → mock. `agent_runs.model` records the provider-qualified id for both real paths; `prompt_version` = `"v1"`. Tokens come from `usage.inputTokens`/`outputTokens`. This is the project's **first functional external dependency**; failure modes (unavailable/rate-limit/timeout/parse-fail) close the run `failed` and surface inline — the operator re-triggers. Observed keeper cost: 679 in / 232 out tokens, ~11 s, ~$0.0055 (Sonnet 4.6).

## K-6.12 — Audit model (agent vs operator)
**Agent actions live in the substrate, not `audit_logs`.** Creating a draft / logging tool calls / decisions is captured by `agent_runs` + `agent_tool_calls` + `agent_decisions` — no `audit_logs` row. **Operator actions on agent output do hit `audit_logs`:** `rewrite_draft.approved`, `rewrite_draft.rejected`, `rewrite_draft.discarded`, `rewrite_draft.published`, and `communication.created`. Audit-inside-vs-outside follows row-count (R-6.7): `createReview` + `publishRewriteDraft` are multi-row → audit inside the txn; `shareNote` / delivery transitions / `discardDraft` are single-row → `writeAuditLog` outside. Phase 9+ cross-cutting audit unions the substrate + `audit_logs`.

## K-6.13 — Worked example (the Job #2 rewriter chain — real, persisted)
**Job #2** (`demo` tenant — HVAC, New York NY, Dispatched). Its `requires_review` note: *"Vendor reported part replacement, $750 over NTE…"*. An operator clicked **Draft client update** → `update_rewriter_v1` ran (real Sonnet 4.6: 679 in / 232 out tokens, confidence **high**) and produced (verbatim):
> *"The technician has identified a part replacement needed to restore cooling on the rooftop HVAC unit. We are currently seeking your authorization before proceeding with the repair. Please expect a follow-up from our team shortly to confirm next steps."*

**Stripped** (verbatim from `agent_decisions.metadata`):
- $750 cost figure
- NTE (Not-to-Exceed) reference and overage detail
- Vendor name (Sunbelt HVAC) — not material to the client-facing update
- Internal authorization workflow framing ("Need to confirm with client before authorizing" reframed neutrally)

**Rephrased:**
- "$750 over NTE. Need to confirm with client before authorizing" → "We are currently seeking your authorization before proceeding" — removes the cost detail while preserving the action required from the client

**Rationale** (verbatim): *"All pricing and internal process language was removed; the core facts (part needed, awaiting authorization, follow-up pending) are preserved in a professional and accurate tone without manufacturing certainty on timing."*

Approved as-is (`edited_content=NULL`) and published. **Persisted chain:** 1 `agent_run` (succeeded, model `anthropic/claude-sonnet-4-6`, prompt_version v1) → 4 `agent_tool_calls` (getJobNote/getJobDetail/listAssignmentsForJob reads + createRewriteDraft write) → 1 `agent_decision` (rewrite_proposal, high, queued_for_review, metadata strippedItems[4]+rephrasings[1]) → 1 `update_rewrite_draft` (published) → 1 `update_rewrite_review` (approve) → 1 `client_update_logs` row → 1 `communication_logs` row (client_portal/client_visible/draft). **Job #2 now has 2 communications:** the earlier shared note (`source_type='job_note'`) + this published update (`source_type='client_update'`), both operable via the delivery machine, both interleaved in the Timeline.

## K-6.14 — What does NOT exist yet (do not claim these)
- **No ad-hoc compose UI / no inbound-logging UI** — `outbound_messages`/`inbound_messages` schema exists; the compose-new + inbound forms are **Phase 6.5** (6e.5).
- **No external transmission** — `delivery_status` is operator-tracked; nothing is actually emailed/SMSed/pushed to a portal. Real send + `email_templates` rendering is **Phase 13**.
- **No per-client rewriter policy** — Phase 6 requires review for **every** draft; per-client `agent_policies` (which clients allow auto-publish) is **Phase 7**. There is no auto-publish in Phase 6.
- **No LLM-native agent tool-use** — the rewriter is a fixed pipeline; an agent that chooses its own tool calls is **Phase 8** (NTE negotiator).
- **No `vendor_update_logs` / `portal_update_queue` activity** — schema-only forward-decls; no writer until Phases 10/12/13. The Phase 6 "update queue" is the rewriter **draft** queue.
- **No `agent_drafts` unification decision** — Phase 6 ships a specialized `update_rewrite_drafts`; shared-vs-specialized is **Phase 7**.
- **No role restriction on review/publish** — any authenticated operator in the tenant can approve and publish (role-gating is Phase 7+).
- **No async / background rewriter** — runs are synchronous (~11 s observed); background runs for bulk/email-triggered rewrites are **Phase 13**.
