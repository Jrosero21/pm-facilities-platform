# Phase 9 — Decisions

The locked calls and their rationale, accumulated across the gates (9a design → 9b schema → 9c readers → 9d seed/harness → 9e dashboard → 9f badge). Format: the decision, then **Why**. Terminology established here (dual-population rule, read-vs-write asymmetry, paired aggregate+single-row reader) is authoritative and flows downstream.

## A. Design-gate principles (9a)

- **Source-agnostic composition.** The dashboard reads jobs / history / billing regardless of intake channel; no metric centers on ServiceChannel or any one source. **Why:** the platform is a multi-channel aggregator (hard rule); analytics that assume a channel would not generalize.
- **Single composed `/dashboard`, not per-role routes.** One canonical landing surface with per-section role-gating, rather than separate role-routed pages. **Why:** one operator often wears several hats; a single surface is cheaper and avoids a post-login role-routing decision. Multi-page split deferred to a later phase if dashboards grow.
- **The action queue is distinct from the `/jobs` inventory.** `/dashboard`'s operational queue is a composite-urgency, top-N *triage* surface; `/jobs` stays the full filterable *inventory*. **Why:** they answer different questions ("what needs attention now" vs "show me all jobs"); conflating them serves neither.
- **"Stalled / aged" is a read-time classification, never a stored status.** Recomputed every render from the threshold map + dwell. **Why:** a stored flag would drift the instant a threshold changes or time passes; the classification is cheap to compute live.
- **Threshold structure = a single constants module** (`stalled-rules.ts`), not a DB table. **Why:** MVP thresholds are fixed; the classifier signature is identical whether thresholds come from a constant map or a table row, so lifting to a tenant-configurable table later costs no refactor.
- **"Lights up as data flows" — build data-blocked metrics now.** Metrics whose source columns (`due_at` / `scheduled_start_at`) are operator-populated and largely NULL today are **built, correct, and rendered with empty states now**, surfacing signal automatically as data accrues. **Why:** roadmap §2.7 — analytics designed from day one; deferring them until data exists would mean a code change later instead of zero-effort activation. (Rule expression in `06-business-rules.md`; chatbot framing in `07-chatbot-knowledge.md`.)
- **Compute-on-read, no materialization.** Every dashboard metric is a live aggregation at render time. **Why:** at current/foreseeable volume the existing `jobs` indexes serve the roll-ups; materialized views / caches are unearned complexity. (The six 9a forks — route, queue, cards, classifier, compute-on-read, retained seed — all resolved on review 2026-05-28.)

## B. Schema gate (9b)

- **The two deferred indexes land in the consuming phase.** `jobs_tenant_due_idx (tenant_id, due_at)` + `jobs_tenant_source_idx (tenant_id, source_type)` were deferred since Phase 4 ("the consumer defines the right composite — add in the consuming phase"); Phase 9 is that consumer. **Why:** the dashboard's overdue detection reads `due_at` and source analytics read `source_type`; adding them speculatively earlier would have guessed the composite shape.
- **Manifest-first → fresh-replay verify discipline (extends CF-8b.1 through `0024`).** The schema gate ran the from-scratch migration replay through the new migration before relying on it. **Why:** re-affirms the Phase-8 CF-8b.1 methodology (a from-scratch rebuild reproduces the live schema) for every added migration.

## C. Reader layer (9c)

- **File-per-domain reader layout mirroring `src/server/billing/`.** Each reader domain is its own module; pure helpers (`stalled-rules`, `percentile`, `resolve-scheduled-start-at`) are separate. **Why:** matches the established Phase-8 billing convention; pure modules stay trivially unit-testable.
- **Explicit `tenantId` first parameter** (not `requireTenant`-internal). **Why:** inherited Phase-8 reader convention — keeps readers callable from any context and tenant-scoping explicit at the call site.
- **Empty-state convention: arrays → `[]`, counts → `0`; never null-for-empty, never throw.** **Why:** the dashboard composes ~9 readers; a well-typed empty result lets every panel render its empty state uniformly ("lights up as data flows").
- **App-side percentile math — documented deviation from "SQL does the work."** SQL computes each interval's duration (`LAG()` + `TIMESTAMPDIFF`) and groups; the reader computes p50/p90/mean in TS via the pure `summarizeSeconds`. **Why:** MariaDB 11.4.10 *does* support `PERCENTILE_CONT`/`MEDIAN` but **window-only** (no plain grouped ordered-set aggregate), which would force a `PARTITION BY` + dedup wrapper; app-side is simpler, testable, trivially extensible. Filter/group stays in SQL.
- **App-side `operationalQueue` tier classification — documented deviation.** The base query returns tenant-scoped open jobs with computed dwell + counts; TypeScript classifies each into the urgency tier (stalled > overdue > unassigned-high-priority > aged), sorts, and slices to N=20. **Why:** the multi-signal precedence rule is more legible and testable in TS than nested SQL `CASE`; volume is small post-filter. Documented so future readers don't refactor it into pure SQL.
- **MariaDB 11.4.10 engine baseline.** Window functions, `TIMESTAMPDIFF`, and the percentile functions' window-only availability are the assumed engine capabilities these decisions depend on.
- **Reader-construction discipline.** When construction surfaces an operationally-correct refinement beyond the literal manifest (e.g. the `is_archived` exclusion, the dual-population rule), **surface it at the sub-batch gate** and fold it back into the manifest before commit — never ship silently nor follow literal text rigidly. **Why:** the manifest is the contract; reality discovered during construction refines it through explicit review.

## D. Seed + harness (9d)

- **Three-stage seed pipeline:** schema replay → global reference seeds (shelled out) → in-process operational seed. **Why:** the operational seed needs the reference vocabulary (statuses/trades/priorities) present first; staging keeps each concern idempotent.
- **Idempotent reset = explicit ordered deletes + `FOREIGN_KEY_CHECKS=0`, not a single tenant-cascade.** **Why (Gate-1 lesson):** a `tenant_id`-FK cascade survey is **necessary but not sufficient** — inter-child RESTRICT FKs (e.g. `jobs.client_location_id → client_locations`) block InnoDB from ordering a single tenant DELETE's children (`ER_ROW_IS_REFERENCED_2`). Future tenant-reset work must survey the inter-child FK graph too, or sidestep it with explicit ordered deletes.
- **User-upsert-by-email** (better-auth users are global, never deleted by the reset). **Why:** preserves user/session records across re-runs while rebuilding tenant membership; also keeps the better-auth NULL-tenant audit rows frozen (no growth) across re-runs.
- **Fixture-derived oracle over hardcoded expected values.** The harness derives expectations from the seed fixture via trivial filters. **Why:** seed + fixture + harness stay co-versioned; a fixture edit updates both seeded data and expectations in lockstep, with no magic numbers to hand-recompute.
- **The retained harness is the project's first standing regression artifact** (partial answer to CF-8c.8.3). **Why:** it elevates the analytics readers from per-sub-batch ephemeral verification to a re-runnable regression check — not a full test framework, but the first persistent one.
- **mysql2 ↔ DB timezone skew is a real bug class.** *"mysql2 client serializes JS `Date` objects in the Node-process timezone, but the MySQL session timezone is `SYSTEM` (DB host's tz). Comparing against server `NOW()` produces a skew equal to the timezone delta between Node host and DB host. Any code writing timestamp values must either (a) use server-anchored expressions like `` sql`NOW() - INTERVAL n SECOND` `` and let the DB compute the value, or (b) write `Date.toISOString()` UTC strings and configure the mysql2 connection's `timezone: 'Z'`. Phase 9 9d uses (a) via the `agoSql` helper, sandbox-scoped. Production is unaffected (real writes use DB-default `CURRENT_TIMESTAMP`). Codebase-wide adoption of (b) at the pool config would eliminate the class but requires auditing existing date-write paths — a future cleanup, not a Phase 9 deliverable."* (Also in `10-known-limitations.md`.)

## E. Dashboard + filter (9e)

- **Read-vs-write role-gating asymmetry (foundational principle).** *"Read-side dashboard panels can extend visibility beyond the corresponding write-side action's role gate when the information is summary-level and management-context-relevant. Action gates remain strict; read gates are calibrated to the operator-management-team mental model."* Concrete: `canSeeFinancials` (read) = `accounting | tenant_admin | super_admin`, while billing **actions** (`enforceAccountingGate`/`isAccountingRole`) = `accounting | super_admin`. **Why:** the write gate exists for segregation of accounting *duties* / preventing accidental mutations — a concern that does not apply to reading three summary counts; mirroring it onto the read side is a category error.
- **Role-gating = generic primitive + named predicates.** `hasAnyRole(ctx, keys)` (the project's first read-side role primitive, `src/server/role-predicates.ts`) with named `canSeeOperations` / `canSeeFinancials` composing over it. **Why:** the primitive is reusable (inheritable by Phase 10/11 portals); named predicates give call-site readability.
- **Palette inheritance is a hard "do not vary per page" constraint.** The dashboard reuses the existing semantic palette (neutral/amber/blue/green/red) via `dashboard/tier-colors.ts`, keyed `Record<UrgencyTier, …>` so adding a tier forces a compile error until its color is assigned. **Why:** consistent color semantics across surfaces is an operator-mental-model requirement; a per-page palette would erode it.
- **Inline panel composition over premature sub-component delegation.** The dashboard composes panels inline, reusing only the shared primitives (`EmptyState`, `tierBadge`). **Why:** Phase-8 pattern; one-off panels don't earn their own components, and inline keeps the composition reviewable in one place.
- **Async `searchParams` (the Next.js 15 `Promise<…>` convention), adopted here first.** `/jobs` receives `searchParams: Promise<{status?, priority?}>` and awaits it; `listJobs` gains additive optional `filters`. **Why:** the framework's current convention; this is the codebase's first searchParams-driven route, establishing the pattern.
- **Implicit-dynamic via `cookies()`; no `force-dynamic`.** **Why:** `requireTenant()`'s `cookies()` read already forces dynamic rendering, so live counts are never statically cached — an explicit directive would be redundant.
- **`EmptyState` + route-level `loading.tsx` established as the project's first shared primitives of their kind.** **Why:** ~9 panels share the empty-state idiom; a route-level loading affordance is the minimal, earned form (per-panel Suspense is unearned at current latency — see `10-known-limitations.md`).
- **Filter indicator = count + clear, not a labeled chip.** **Why:** the count form honors the pinned IDs-only `resolveJobsFilters` contract and avoids extra label-lookup queries; operators arrive via a card click and need the active-filter signal + an escape hatch, not a re-statement. (Labeled chip = a future UX refinement, `closeout-carryforwards.md`.)
- **Count-in-heading for list/table sections** ("Operational queue · N jobs"). **Why:** gives a scanning operator an immediate quantitative anchor, parallel to the stalled summary's prominent count (a 9e.6 ergonomics finding).

## F. Job-detail badge (9f)

- **Paired aggregate + single-row reader pattern.** `isJobStalled` is the single-row counterpart to `countStalledJobs` — same predicate, same subqueries verbatim, scoped to one job. **Why:** guarantees the job-detail badge classifies *identically* to the dashboard queue (no second code path to drift), and avoids the "cycle the all-job aggregate to classify one job" anti-pattern. The pattern: extract the predicate (`isStalled`, 9c.6) → build the aggregate reader (`countStalledJobs`, 9c.5) → add the single-row reader (`isJobStalled`, 9f) only when a consumer surfaces.
