# Phase 7 — Chatbot Knowledge

Source-of-truth for how AI-assisted scope generation, the generic agent-config substrate, and the second production agent work after Phase 7. Written to stand alone: an LLM with only this file should answer operational questions correctly. Cross-references `02-decisions.md`, `06-business-rules.md`, `08-db-changes.md`, `09-api-routes.md` but does not depend on them. Builds on Phase 6 chatbot knowledge (the agent substrate, the update rewriter, §2.9).

## K-7.1 — What Phase 7 adds
Three things. **(1) AI-assisted scope generation:** an operator turns a job's short problem description into a structured, reviewable **scope of work** (ordered technician steps) — generate → review/edit → approve → publish, all on the job page. **(2) A generic agent-config substrate:** DB-stored, versioned **prompts** (`ai_prompt_templates`) and **policies** (`agent_policies`), per-tenant with platform **defaults**, replacing Phase 6's in-code prompt and inline policy literal — every agent now resolves prompt + policy at runtime. **(3) The second production agent** `scope_generator_v1`, built on the Phase 6 agent substrate (the same `agent_runs`/`tool_calls`/`decisions` + runner). The Phase 6 update rewriter was **retrofitted** onto the same config substrate, so both agents share one pattern. Everything stays **draft-then-review-then-publish** (§2.9): the agent never writes operational state.

## K-7.2 — The agent-config substrate (prompts + policies)
Two tenant-scoped tables + two platform-default siblings:
- **`ai_prompt_templates`** (tenant) / **`ai_prompt_template_defaults`** (global, no `tenant_id`): `agent_id`, `variant` (default `'default'`), `version`, `status` enum(`draft`/`active`/`archived`), `system_prompt`, `user_prompt_template` (Mustache, may be NULL → agent assembles in code), `model_hint`, `temperature`.
- **`agent_policies`** (tenant, nullable `client_id`) / **`agent_policy_defaults`** (global): `policy` json, `version`, `status`. Phase 7 exercises only `{requiresReview:true}`.
- **Resolution** (uncached in Phase 7): prompts `(tenant, agent, variant)` → `defaults(agent, variant)`; policies `(tenant, client, agent)` → `(tenant, agent)` → `defaults(agent)`. **Prompts fail closed** (no active row → `NoActivePromptError`, run fails); **policies fail safe** (no match → `{requiresReview:true}`). Both `JSON.parse` json columns at the read boundary (R-6.19). (R-7.3.)
- **Single-active is a write-path invariant (R-7.1):** at most one `active` row per resolver key, enforced in `activatePromptTemplate`/`activateAgentPolicy` (demote-then-promote in a txn with affected-row assertions), not by a DB unique (the nullable `client_id` rules one out on policies).

## K-7.3 — The scope generator (`scope_generator_v1`)
The second production agent. Given a job's **problem description + client/location/trade/priority** (read via `getJobDetail` — current-job context only, no history/templates/RAG), it produces a **structured scope**: an ordered list of steps, each with an instruction, an optional `category` (assess/perform/cleanup/verify/document) and an optional `expectsPhoto` flag, plus `assumptions[]`, `confidence`, and a one-line `rationale`. It is a **fixed pipeline** (read-broad/write-narrow, like the rewriter): one read tool `getJobDetail`, one write tool `createScopeDraft` (a draft at `pending_review`). It **always queues for review** (§2.9) — disposition `queued_for_review`, regardless of policy (Phase 7 has no auto-execute path). Trigger is **operator-manual only** (no auto-generate on job creation). Re-running creates a new draft.

## K-7.4 — The scope draft lifecycle
`job_scope_drafts.status` mirrors the rewriter's draft vocab **1:1**: `pending_review` → `approved` → `published` (terminal); `pending_review`/`approved` → `discarded`; `pending_review` → `rejected`. The AI's steps live in `proposed_steps` (JSON, **immutable** — the "what the AI produced" audit). The operator reviews (sees assumptions, rationale, confidence, the editable step list) and **approves** (their edited list is stored on `job_scope_reviews.edited_steps`, NULL if unchanged), **rejects** (with a required reason), or **discards**. Effective published steps = **`edited_steps ?? proposed_steps`**. Edit affordances: reorder, add, remove, rewrite, change category, toggle expectsPhoto. Approving with an empty list or any empty-instruction step is refused (`SCOPE_DRAFT_REQUIRES_STEPS`).

## K-7.5 — Publish (the only path to `job_scope_steps`)
`publishScopeDraft` is the **sole writer** of `job_scope_steps` (R-7.2) and refuses anything not `approved`. In one parent-before-child txn (lock job → lock draft → re-check `approved` → **gate** → write): it inserts N `job_scope_steps` rows (`source='edited'` if the operator edited, else `'ai_generated'`; contiguous `step_order`), writes both `jobs` text columns (K-7.6), sets `jobs.scope_generation_status='approved'`, advances the draft to `published`, and writes the `scope_draft.published` audit row inside the txn. The **gate (R-7.2 / L-7.7):** if the job already has active `job_scope_steps`, publish throws **`ScopeAlreadyPublished`** — one scope per job in Phase 7; re-scope is deferred.

## K-7.6 — The two-column model (generated vs approved)
`jobs.generated_scope_of_work` = flat-render of the draft's immutable `proposed_steps` (the **raw AI artifact**); `jobs.approved_scope_of_work` = flat-render of `edited_steps ?? proposed_steps` (the **operator-approved final**). Both are written **at publish** (the agent writes neither). They are **identical on a no-edit publish, and diverge on an edited publish.** The flat columns are **derived views** (numbered, instruction-only) for consumers that can't read structured data; the **canonical** record is `job_scope_steps` — never reconstruct structure by parsing the text (D-7.2 / D-7.3).

## K-7.7 — The rewriter retrofit (now on the substrate)
`update_rewriter_v1` no longer carries an in-code system prompt or an inline policy literal — it resolves both from the substrate (`resolveActivePrompt`/`resolveAgentPolicy`), exactly like the scope generator. Its system prompt now lives in `ai_prompt_template_defaults` (seeded from the former `prompt.ts` constant, byte-equal); `buildUserPrompt` stays in code (`user_prompt_template` NULL). Behavior is **unchanged** vs the Phase 6 keeper run. One visible change: `agent_runs.prompt_version` records **`"1"`** (the DB version) for post-retrofit rewriter runs, where it was `"v1"` before — a deliberate provenance signal of DB-resolution, not a regression (D-7.4). The shared `resolveAgentRouting` (extracted in the retrofit) serves both agents.

## K-7.8 — The operator UI (the Scope of work section)
On the job page, between *Problem description* and *Dispatch*. **Generate scope** button (hidden once a scope is published — D-7.6). A three-group draft queue mirroring the rewriter's: **Pending review** (expand to review/edit/approve/reject/discard), **Ready to publish** (approved drafts → Publish button; or, when the job is already published, the gate note + a **Discard** control), **Dismissed** (rejected/discarded, collapsible, status-labelled). Published scope renders as a numbered list. Published-state and pending-state are **derived from the substrate tables** (`job_scope_steps` / `job_scope_drafts`), not from `scope_generation_status` (D-7.5). A gated approved sibling can be **discarded** (D-7.8) — the disposal path the gate note promises.

## K-7.9 — LLM routing
`resolveAgentRouting(opts)` (shared): mock if the agent's mock var **or** the global `AGENT_MOCK` is `"1"`; else gateway (`AI_GATEWAY_API_KEY`, model string `"anthropic/claude-sonnet-4-6"`); else direct (`ANTHROPIC_API_KEY`, `"claude-sonnet-4-6"`); else mock. Scope-gen knobs: `SCOPE_GEN_MOCK` / `SCOPE_GEN_MODEL`; rewriter: `REWRITER_MOCK` / `REWRITER_MODEL`. `agent_runs.model` records the provider-qualified id; structured output via AI SDK v6 `generateObject` + Zod. (D-7.1; R-6.25 inherited.)

## K-7.10 — Worked examples (real, persisted)
- **Job #1 — the scope keeper (no-edit publish).** "Toilet clog…" → `scope_generator_v1` (real Sonnet 4.6) produced a **9-step** scope (assess→perform→verify→cleanup→document; photos on assess/verify/document; no standalone "upload photos" step). Approved with no edits → published. `job_scope_steps` = 9 rows `source='ai_generated'`; `generated_ == approved_`.
- **Job #2 — the edited-publish artifact.** "Rooftop HVAC not cooling…" → a **14-step** draft. The operator applied every edit affordance (reorder, rewrite, category change, expectsPhoto toggle, add, remove) → **8** published steps `source='edited'`. `generated_scope_of_work` (flat of the 14 proposed) **≠** `approved_scope_of_work` (flat of the 8 edited) — the two-column divergence, verified.

## K-7.11 — What does NOT exist yet (do not claim these)
- **No scope templates in use** — `scope_templates`/`scope_template_steps` are empty schema; no seed, no UI, no grounding (OQ #2). Template-driven generation is future.
- **No auto-trigger** — scope generation is operator-manual only; nothing generates on job creation.
- **No re-scope** — once a job has a published scope, the trigger is hidden and `publishScopeDraft` rejects a second publish (`ScopeAlreadyPublished`, L-7.7). Replacing a published scope is a future workflow.
- **The policy resolver is inert** — it resolves and is load-bearing for future phases, but Phase 7 ships only `{requiresReview:true}`, so disposition is always `queued_for_review`; no auto-execute branch (L-7.1, L-7.5).
- **`jobs.scope_generation_status='pending_review'` is reserved-unused** — the status goes `not_started → approved`; pending state is read from `job_scope_drafts` (L-7.6 / D-7.5).
- **No historical-scope grounding / cross-job learning / RAG** — current-job context only (OQ #6); overlaps Phase 16.
- **No role restriction** — any authenticated tenant member generates/approves/publishes (role-gating is future).
- **No async / background generation** — runs are synchronous; bulk/triggered generation is future (Phase 13/16 territory).
- **Per-step edit provenance is whole-set** — an edited publish marks **all** steps `source='edited'`, not per-step (L-7.3).
