# Phase 7 — Business Rules

Rules introduced in Phase 7, each with the reasoning behind it. Inherits Phase 0–6 rules — notably the **§2.9 agent-under-policy** posture, the agent substrate's immutable-audit tables (R-6.11) and audit split (R-6.12, agent actions in the substrate / operator actions in `audit_logs`), `agent_id = {name}_v{major}` versioning (R-6.16), registry `testOnly` (R-6.20), the agent-has-no-path-to-operational-state invariant (R-6.15), parent-before-child publish lock order (R-5.7 / R-6.21), audit-inside-vs-outside by row count (R-6.7), the MariaDB `json()` read-parse rule (R-6.19), short FK-prefix-under-64-chars (R-6.22), and the literal-acceptance-criterion closeout review (R-6.23). Phase 7 **generalizes the §2.9 substrate** into reusable per-tenant configuration (DB-stored prompts + policies) and adds the platform's **second agent** (`scope_generator_v1`) plus its publish substrate (`job_scope_steps`).

## R-7.1 — Single-active config is a write-path invariant (not a DB constraint)
- On `ai_prompt_templates` and `agent_policies`, **at most one row per resolver key may have `status='active'` at any time** (key = `(tenant_id, agent_id, variant)` for prompts; `(tenant_id, client_id, agent_id)` for policies). The DB gives only **partial** coverage: `ai_prompt_templates` has `UNIQUE(tenant_id, agent_id, variant, version)` (blocks duplicate *versions*, not duplicate *actives*); `agent_policies` has **no** unique at all, because the nullable `client_id` + MariaDB's NULL-as-distinct semantics make one unreliable (two `(tenant, NULL, agent)` rows would not collide). The invariant is therefore enforced in the **data-layer activate functions** (`activatePromptTemplate` / `activateAgentPolicy`): in one txn, demote the current active for the key (no `LIMIT`) and assert the demoted count ≤ 1 (`SingleActiveInvariantViolated`), then promote the target and assert affected == 1 (`ActivationTargetMismatch`). The demote's `WHERE status='active'` excludes the new value, so the affected-row count is **driver-mode invariant** (identical under default mysql2 and `CLIENT_FOUND_ROWS`). Read resolvers add `ORDER BY version DESC LIMIT 1` as a **non-load-bearing tie-break** safety net.
- **Dec-3b nuance:** the guarantee is **invariant-preservation under concurrent activation, not loser-detection.** Phase 7 ships serialized-write **last-writer-wins** (two concurrent activations both serialize on the row locks; exactly one ends active; both may report success). A future phase introducing high-concurrency activation paths (automated tenant onboarding, bulk policy migration) should reconsider a loser-detection upgrade (optimistic version check returning a typed error to the loser).
- **Why:** this is the largest constraint downgrade in the phase — on `agent_policies` single-active is 100% code with **no DB safety net** — so naming it makes the obligation explicit and prevents a future contributor from "fixing" the perceived driver dependency with a count-then-update (which would open a within-txn race the current shape doesn't have). The closeout literal-acceptance review (R-6.23) must demonstrate the write path **rejects or atomically supersedes** a second activation at an existing key. (Decision B / 7b; F3.)

## R-7.2 — `job_scope_steps` has exactly one writer: `publishScopeDraft`
- The canonical, structured scope of a job is `job_scope_steps`, and the **only** code that writes it is the human-gated `publishScopeDraft` (in one parent-before-child txn). The agent (`scope_generator_v1`) writes **only** `job_scope_drafts` at `pending_review` — it has no path to `job_scope_steps` or the `jobs` columns (R-6.15 generalized to the second agent). The operator UI derives a job's **published-scope state** from `job_scope_steps` presence (D-7.5), so that derivation depends on this single-writer invariant.
- **Why:** §2.9 — AI output is never final until reviewed; making `job_scope_steps` write-reachable only through the gated publish action makes the invariant structural, not conventional. Any **future** writer (a re-scope workflow, a data migration, a repair script) must either reach the same UI-visible state semantics as publish, or the UI derivation must tighten its query. (Generalizes R-6.15.)

## R-7.3 — Config resolution fails CLOSED for prompts, SAFE for policies; mock skips both
- **Prompts fail closed (real runs):** `resolveActivePrompt(tenant → default)` that finds no active row **throws `NoActivePromptError`**, closing the run `status='failed'`. It does **not** fall back to a code constant — a stale code prompt would make `agent_runs.prompt_version` lie about what ran.
- **The mock path is a third branch, outside the closed/safe dichotomy (Dec-1):** under mock routing the agent **skips prompt resolution entirely** and records `prompt_version='mock'`, so fail-closed never fires in mock — dev must never hard-fail on a missing key. Mock neither resolves a prompt nor depends on a policy disposition.
- **Policies fail safe:** `resolveAgentPolicy((tenant,client,agent) → (tenant,agent) → defaults(agent))` with no match returns the literal **`{ requiresReview: true }`**. Absence of a policy must **never** mean auto-execute.
- **Why:** the two failure modes are deliberately opposite because their risks are opposite — a missing prompt should *stop* (provenance integrity over convenience), a missing policy should *protect* (§2.9 over availability). Both resolvers `JSON.parse` json columns at the read boundary (R-6.19). (7a Surfaces #2/#3; D-7.7 enforces the policy disposition in the data layer.)
