# Phase 8 вЂ” Decisions

The locked calls and their rationale, accumulated across the gates (8a design в†’ 8b schema в†’ 8c construction). Format: the decision, then **Why**. Open-question resolutions cite their OQ; data-layer locks cite their sub-batch.

## A. Design-gate principles (8a)

- **R-7.1 вЂ” single-active write-path, no DB unique.** "At most one active X" invariants (active NTE rule per tuple; single live proposal revision per chain) are enforced by the **writer** (demote-then-assert / lock-the-root), not a DB unique constraint. **Why:** the "active" set is a moving target the writer already serializes under a lock; a DB unique on a status column is brittle (partial-index gymnastics) and wouldn't express the demote-on-supersede semantics.
- **R-7.2 вЂ” single-writer for mutable substrate.** Each mutable money column has exactly one canonical writer; agents/automation have no path to it. **Why:** money correctness depends on every mutation flowing through the same rounding + event-emitting code; multiple writers = drift.
- **D-7.3 вЂ” published scope is derived, isolated.** Billing never writes the published-scope substrate (`job_scope_steps` / `jobs.approved_scope_of_work`); proposals/COs carry their own independent `scope_snapshot` / `scope_delta_snapshot` text. **Why:** a proposal's scope is a quote artifact, not the operational scope; conflating them would let a billing action mutate operational state.
- **D-7.7 вЂ” invariants at the data-layer boundary.** Cross-field invariants with no clean DB constraint (the payment XOR) are enforced in the writer at the boundary. **Why:** MySQL CHECK support is unreliable across the MariaDB target; the writer is the honest enforcement point.
- **OQ-1 вЂ” money is `decimal(12,2)`, round-each-line-then-sum, round-half-up.** **OQ-2 вЂ” same-currency MVP** (`USD` everywhere; per-record currency stored but not converted).
- **8c-D2 вЂ” accounting gate = `accounting` role OR `super_admin` auto-pass; NO `tenant_admin`.** **Why:** issuing/paying/closing money is a distinct control from operating a job; super_admin is the platform escape hatch.
- **8c-D4 вЂ” line shape = shared base + AR markup extension.** All four line tables share `baseLineItemColumns()`; the three AR tables add `arMarkupColumns()`; vendor (AP) lines carry no markup. Cost-basis + uplift (`extended = qtyГ—unit_price`, `markup = round(extendedГ—pct/100)`).
- **8c-D6 вЂ” billing close is a soft "ready" signal, not a hard precondition.** `markBillingClosed` never consults readiness; the operator decides.
- **OQ-27 вЂ” defer ALL agents in Phase 8.** No NTE negotiator; L-7.1 resolver stays inert; Q-7.1 (agent-config seed split) carries forward. **Why:** В§5.4 stay-in-phase вЂ” roadmap В§8 has no agent deliverable; substrate + data must exist first.

## B. Schema decisions (8b)

- **8b-D1 вЂ” `emergency_nte_multiplier` lives on `client_billing_rules`** (per-client override; NULL = a tenant-default of 1.50 by design), not a new tenant-config table. **Why:** it's per-client billing policy; the column reuses the existing client-side config substrate. _(The column ships, but the resolver that would apply it was never wired in Phase 8 вЂ” stored-but-inert; see `10-known-limitations.md`.)_
- **`source_external_id` has no uniqueness** (vendor invoices) вЂ” inbound dedup is a Phase-13 ingestion concern.
- **Parallel approval shape** вЂ” `proposal_approvals` and `change_order_approvals` share the `{accepted, declined}` decision enum (8b В§3) в†’ see CF-8c.6.1.
- **8 migrations (0016вЂ“0023)** applied as one schema gate before construction; see `08-db-changes.md`.

## C. Data-layer decisions (8c.1вЂ“8c.10)

- **8c.1 (NTE):** the resolver ladder is location в†’ client-wide в†’ handyman-location в†’ handyman-client-wide; the handyman trade code is `'HANDY'` (not `'HANDYMAN'`). Single-active via demote-then-assert (R-7.1). `archiveClientNteRule` was a silent state change at 8c.1 в†’ audited at 8c.11e (CF-8c.1.1, closed).
- **8c.2 (totals):** **explicit-mode rounding** вЂ” every `Big.round` passes `Big.roundHalfUp` explicitly; never the mutable global `Big.RM`. **Why:** another module could flip the global to banker's rounding and silently change money. `computeMarkup` order is `(extended Г— pct) / 100`.
- **8c.3 (events):** `emitJobBillingEvent` is THE enforcement boundary вЂ” validates the 21-type taxonomy + field shapes, runs inside the caller's txn. **`change_order.created` dropped** (first CO emit is `change_order.submitted` вЂ” drafts are transient); **`vendor_invoice.created` dropped** but **`client_invoice.created` kept** вЂ” the AR document's *authoring* is a meaningful operator state; the AP invoice's *receipt* is. 0-to-many record refs, no XOR on the event.
- **8c.4 (createJob NTE):** `createJob` is the **sole writer of `jobs.not_to_exceed_amount`** (the first Phase-4в†’Phase-8 dependency, one-way acyclic). 5-case override matrix; override = Case C only (a rule resolved AND the operator value differs) в†’ `nte.overridden` audit event. No `Big` in `jobs.ts` (plain `===` on canonical strings вЂ” trust-at-boundary). в†’ CF-8c.4.1 (multi-currency comparison deferred).
- **8c.5 (proposals):** `LIVE_STATUSES` (revision-chain occupancy: draft/sent/viewed/accepted) vs `WITHDRAWABLE_STATUSES` (draft/sent/viewed). **Accepted is live-but-not-withdrawable** вЂ” a commitment; you revise or change-order, you don't withdraw it. Revision chain serialized by locking the chain ROOT `FOR UPDATE`.
- **8c.6 (change-orders):** COs are **forward deltas, no revision concept** (8c-D5) вЂ” a redo is a new CO. **`getEffectiveNte` computed-on-read** (OQ-14): base + ОЈ approved-CO totals; returns `null` when the base is null. approve maps to `{status:"approved", decision:"accepted"}` (the shared enum) в†’ CF-8c.6.1.
- **8c.7 (vendor invoices / AP):** **column-vs-event split** вЂ” `recalculateVendorInvoiceTotals` sets the `exceeds_nte` column on record/edit (always truthful), but the `nte.exceeded` **event** fires only at approve (the commit-to-pay). **Two independent breach events** (per-invoice dispatch/job + job-aggregate first-crossing). `governingNte` is **passed into** recalc by the caller в†’ keeps `totals.ts` cycle-free + event-free. `approveVendorInvoice` `FOR UPDATE`-locks the job row for correct first-crossing under concurrency (lock order: invoice в†’ job).
- **8c.8 (client invoices / AR):** **markup three-way** вЂ” `undefined` в†’ snapshot the default `client_billing_rules` markup; `null` в†’ explicit zero; a value в†’ override (snapshot-at-creation, not live). Payment-terms snapshot at header creation. **AR revenue = `status='sent'` only** (there is no `paid` status; `paymentStatus` is orthogonal). **Option 2 role-gate** вЂ” extract the pure predicate `isAccountingRole` (no test framework exists, CF-8c.8.3) so the policy is unit-testable; the action IS the live enforcement. Void requires `status='sent'` (drafts can't be voided) в†’ CF-8c.8.2.
- **8c.9 (payments):** **XOR invoice ref** (exactly one of vendor/client, agreeing with `direction`; D-7.7). **Writer-derived `job_id` via type-absence** вЂ” `RecordPaymentInput` has no `jobId` field, so the compiler forbids forwarding it (the strongest form of R-7.2). **`payment_status` sole-writer** (`payments.ts` owns unpaidв†’partially_paidв†’paid). **Direction-discriminated single-sided writer** вЂ” two disjoint branch functions, each touching one invoice table; deliberate duplication for grep-provable ARв†”AP non-coupling. Overpayment allowed (ОЈ>total в†’ paid) в†’ CF-8c.9.1. paid-event on first crossing only.
- **8c.10 (billing close):** the **first billing writer into the operational lifecycle** вЂ” writes `jobs.current_status_id` + `closed_at` + `job_status_history` + `job_events` + `billing.closed` + `audit_logs`, atomic. **Replicates the Phase-4 status-transition dual-write inline** via schema imports (never imports `@/server/jobs`) вЂ” the existing pattern is logic-free inserts, nothing to drift. The **narrowed sole-writer guarantee** вЂ” `.update(jobs)` is now expected, but `not_to_exceed_amount` is never touched. `closed_at = COALESCE(existing, now())` (first-close-wins). Billing close is **independent of operational status** (transitions from any status; only guard is already-`CLOSED_BILLED`). `finalMargin` snapshot computed before the txn.

## D. UI + action decisions (8c.11)

- **Two coexisting role-check patterns** (established here, validated across ~35 wrappers): (1) `requireRole(role)` directly for simple gates; (2) `requireTenant()` + an extracted pure predicate for gates that benefit from independent unit-testing. Phase 8 uses (2) for all four accounting gates via `enforceAccountingGate(ctx)` (8c.11d), the `isAccountingRole` policy in `auth-context.ts`.
- **The action-reshape template** (8c.11b, copied by 11c/11d/11e): `useActionState<State, FormData>`; **redirect-vs-error split** вЂ” auth/tenant failure в†’ redirect (backstop), F3 operational error в†’ inline `{ error }`; **catch discipline** вЂ” a specific allowlist (instanceof F3 + exact sentinels + the `INVALID_LINE_` prefix) в†’ inline, everything else `throw e` (never a bare `e.message` swallow вЂ” operational states surface, programmer errors propagate); create/revise в†’ redirect to the new record.
- **Action file domain-scoping** вЂ” per-record-type actions co-locate with their record (`proposals/actions.ts`, вЂ¦); cross-domain/job-level actions stay in `billing-actions.ts` (payment touches AR+AP via the XOR; close touches the job lifecycle).
- **`enforceAccountingGate(ctx)` extracted** (8c.11d) вЂ” takes `Pick<AuthContext, "roleKeys" | "isSuperAdmin">` (narrowed; depends only on what it reads); the gate is centralized + structurally verifiable. **Defense-in-depth:** gated buttons render only when `isAccountingRole(ctx)` is true; the action is the backstop.
- **Markup three-way UI** вЂ” two operator-reachable paths (blank в†’ snapshot, `"0"` в†’ explicit zero) + one programmatic-only path (explicit `null` via a direct data-layer call). No separate "no markup" checkbox needed.
- **Vendor-invoice creation is assignment-anchored** (CF-8c.11d.1) вЂ” `vendorId` + `assignmentId` derive from the operator's dispatch selection; no free vendor picker.
- **Five UI data-layer touches**, each a reader-only addition where a screen needed it (or, for 11e, the CF-8c.1.1 audit): the `listJobBillingEvents` actorName LEFT join (11a), the four line-item list readers (11b/11c/11d), the NTE audit inserts (11e). Each gated with its own mini-verify; the UI consumes the verified data layer, it doesn't extend it silently.

## E. Architectural patterns established (for future phases)

- **Big.js explicit-mode rounding** is the project money standard.
- **Computed-on-read for derived financial truth** (effective NTE, margin, readiness) вЂ” avoids stored-value drift.
- **Type-absence as a sole-writer guarantee** (payments' `job_id`) вЂ” the compiler enforces the invariant.
- **Pure-predicate authz extraction** for testability where the framework gate can't run in the test harness.
- **Ephemeral verify scripts** as the empirical test layer (CF-8c.8.3 вЂ” no standing test framework yet).
